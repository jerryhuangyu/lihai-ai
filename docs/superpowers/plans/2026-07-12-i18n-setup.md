# i18n Setup Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 導入 argus 風格的 i18n（react-i18next），並全量遷移元件層 hardcoded 字串到 en/zh-TW 雙語。

**Architecture:** `src/i18n/config.ts` 建立全域 i18next instance（`react-i18next` + browser LanguageDetector），5 個 feature-aligned namespace（common/shell/import/dashboard/sessions）。元件以 `useTranslation` 取字串；純函式 option builder 不碰 hook，改由 Card 端把 `labels` 物件當參數傳入。

**Tech Stack:** i18next ^26、react-i18next ^17、i18next-browser-languagedetector ^8、@lingual/i18n-check ^0.9（dev）；既有 Vite + React 19 + TS strict + Vitest。

> **狀態：As-built（已完成）**。branch `feat/i18n-setup`，commits a5db69a..8168f36（7 tasks + 1 fix wave）。與計劃的差異：Task 2 另加 `@testing-library/user-event` devdep（新 toggle 測試用）；Task 3 navItems `title`→`titleKey` 連帶改 app-header；Task 5/6 tooltip 走 `{{var}}` labels 代入、marker fixture TDD；fix wave 把 EXPORT_HINT 搬進 `import` ns、抽共用 `src/features/charts/fillTemplate.ts`。最終 129 tests + build + i18n:check 全綠全清。

## Global Constraints

- 型別檢查用 `pnpm build`（`tsc -b`），不要用裸 `tsc --noEmit`。
- 套件管理用 `pnpm`，不用 npm。
- 純函式（parser/join/aggregate/option）不呼叫 hook、不呼叫 `Date.now()`；本計劃 option builder 只能透過參數取得字串。
- Resource key 用 `en` / `zh`；detector 可能回報 `zh-TW`，判斷語言用 `startsWith('zh')`。
- `fallbackLng: 'en'`；`en` JSON 必須完整正確。
- 註解一律繁體中文、無 emoji。
- 每個 task 結束 commit（Conventional Commits，結尾附 `Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>`）。
- 分支：`feat/i18n-setup`（已存在，勿碰 main）。
- 範圍邊界：**只做元件層**。排除 `src/parsers/`、`src/workers/`、`src/store/`、`src/export/`、`src/import/*.ts`、`src/ui/format.ts` 格式化邏輯。
- 每個字串搬移：zh-TW JSON 逐字保留現有中文；en JSON 人工撰寫英文；key 用巢狀語意命名（如 `dailyCost.series`），不用整句當 key。

- [ ] **Step 4: 建立 config.ts**

```ts
import i18next from 'i18next'
import LanguageDetector from 'i18next-browser-languagedetector'
import { initReactI18next } from 'react-i18next'
import { enTranslation } from './en'
import { zhTWTranslation } from './zh-tw'

export const defaultNS = 'common'
export const resources = {
  en: enTranslation,
  zh: zhTWTranslation,
} as const

// 純 client SPA（Vite + GitHub Pages），無 SSR，window 恆存在，
// 故不需要 argus 的 isBrowser 守衛。
const i18n = i18next.createInstance()

i18n
  .use(initReactI18next)
  .use(LanguageDetector)
  .init({
    fallbackLng: 'en',
    debug: import.meta.env.DEV,
    defaultNS,
    ns: ['common', 'shell', 'import', 'dashboard', 'sessions'],
    resources,
    interpolation: { escapeValue: false },
  })

export default i18n
```

### Migration tasks 共通流程（Task 3–7 皆適用）

每個遷移 task 對指定檔案做同一機械轉換：

1. 讀該檔，找出所有**使用者可見**的 hardcoded 中文字串（JSX 文字、`aria-label`、`title`、placeholder、傳給 option 的 label 等）。排除註解。
2. 在對應 namespace 的 `zh-tw/<ns>.json` 逐字寫入原中文，key 用巢狀語意命名；`en/<ns>.json` 寫對應英文，key 結構完全一致。
3. 元件內 `const { t } = useTranslation('<ns>')`，字串換成 `t('key')`；帶變數用 interpolation `t('key', { count })`。
4. 更新該檔既有 `*.dom.test.tsx` 中對中文文案的斷言為 i18n 後預期（測試環境已由 Task 1 初始化 i18n；DOM 測試預設語言為 detector 於 jsdom 的結果，若不穩定則在 test `beforeEach` 呼叫 `await i18n.changeLanguage('en')` 後斷言英文）。
5. 移除該 namespace JSON 的 `_` 佔位。
6. `pnpm test <相關測試>` PASS、`pnpm build` 通過、`pnpm i18n:check` 無缺漏後 commit。

**判斷語言穩定性**：所有既有 `*.dom.test.tsx` 在斷言前一律 `await i18n.changeLanguage('en')`，斷言英文文案，確保 CI 穩定。
